Act as an expert software engineer.
Review the provided git diff (or list of changes) and generate a pull request summary consisting of a top-level Conventional Commit message followed by a detailed breakdown.

Strictly follow these rules for the Top Summary:
1. Format must be: <type>(<optional-scope>): <short description in imperative mood>
2. Choose the most accurate type from: feat, fix, docs, style, refactor, perf, build, ci, chore, or test.
3. If the changes introduce a breaking change, append an exclamation mark after the type or scope (e.g., feat!: or feat(auth)!:).
4. Keep the summary under 72 characters, lowercase the description, and do not end it with a period.
5. Avoid overly verbose descriptions or unnecessary details in this line.

Strictly follow these rules for the Detailed Description below the summary:
1. Provide a detailed, bulleted description of what the changes actually achieve.
2. Prioritize newly added files over edited existing files, highlighting the new architectural components first.
